# 第 7 课：Workflow（工作流编排）

> 跑：`npm run workflow`

## ★ 面试必考：Workflow vs Agent
| | **Workflow** | **Agent** |
|---|---|---|
| 控制流谁决定 | **你的代码**写死 | **LLM 自己**决定下一步 |
| 路径 | 可预测、可测试 | 动态、不可预测 |
| 成本/延迟 | 低、稳定 | 高、不定 |
| 适合 | 流程清晰的任务 | 开放、需临场决策的任务 |

> **经验法则：能用 workflow 就别上 agent。** 先把流程写死，确实需要动态决策时再把控制权交给 LLM。
> 二者是连续谱：workflow 里某个节点可以是一个小 agent；agent 内部也常嵌固定 workflow。

一句话记法：**Workflow 是「LLM 填空」，Agent 是「LLM 主导」。**

## 五大模式（来自 Anthropic《Building Effective Agents》）
| 模式 | 干什么 | 本仓库 |
|---|---|---|
| **1. Prompt Chaining** | 拆成串行步骤，步骤间可插「代码门」校验 | 本课 案例 1 |
| **2. Routing** | 先分类，再分发到不同分支 | 本课 案例 2 |
| **3. Parallelization** | 拆成互不依赖的子任务并行 | 第 5 课 `demo`/`dag` |
| **4. Orchestrator-Workers** | 主控动态拆子任务派给 worker，再汇总 | 见下「延伸」 |
| **5. Evaluator-Optimizer** | 生成器 + 评审器闭环，不达标就改 | 本课 案例 3 |

## 本课三个案例（`workflow.ts`）
### 案例 1：Prompt Chaining + Gate（串行 + 质量门）
```
需求 →①LLM 生成大纲→【代码门：大纲≥3条?】→②LLM 按大纲扩写
                          └ 不过就中止，不浪费下一次调用
```
要点：**「门」是你写的确定性 `if`，卡在两个 LLM 步骤中间。**
把大任务拆成「先列提纲、再扩写」，每步更简单、更可控，还能在中间做质量校验。

### 案例 2：Routing（路由分流）
```
用户消息 →①LLM 分类(退款/技术/其他) →② 路由到对应 handler
```
要点：**分类与处理解耦。** 每个分支可用不同 prompt、不同工具、甚至不同模型
（便宜模型做分类，贵模型只处理难分支 → 省钱）。

### 案例 3：Evaluator-Optimizer（评审-优化循环）
```
①生成器产稿 →②评审器打分+给意见 →③不达标带着意见重写 → 循环
                                        └ 终止：评分达标 或 到最大轮数
```
要点：**两个 LLM 角色（写手 + 评委）形成闭环**，用「阈值 + 最大轮数」做确定性终止，避免无限循环。
适合「有明确好坏标准、且反馈能指导改进」的任务（文案、翻译、代码）。

## 延伸：Orchestrator-Workers（本课未单独写）
和 Parallelization 的区别：**并行是子任务【固定】，编排者是子任务【由 LLM 动态拆】。**
主控 LLM 看任务后决定拆几个、拆成啥（写代码时改哪几个文件事先不知道），分派 worker，再汇总。
——这一步就开始带 agent 味了（控制流部分交给了 LLM）。

## 工程要点（面试加分）
- **幂等与状态**：长流程要能断点续跑 → 每步结果落库，失败从断点恢复（参考 LangGraph 的 checkpoint）。
- **Human-in-the-loop**：高风险节点（退款、删库）插人工审批门，本质就是案例 1 的「门」换成等人。
- **可观测**：每个节点埋 trace（见第 4 课），失败能定位是哪一步、哪个 prompt。
- **确定性终止**：循环类（案例 3、ReAct）必须有 `max_rounds` 兜底，否则可能烧爆 token。

## 面试问答速记
**Q：什么时候用 workflow，什么时候用 agent？**
A：任务路径**事先能画出流程图**→workflow（可预测、可测、便宜）；任务**开放、每次步骤都不同、需临场决策**→agent。
先用 workflow，挡不住了再上 agent。

**Q：你做过的 RAG 问答属于 workflow 还是 agent？**
A：固定 RAG 流水线（检索→生成）是 workflow；若让 LLM 自己决定「要不要检索、检索几轮、换什么 query」就成了 agent（Agentic RAG）。

**Q：怎么防止 agent/循环失控？**
A：① 最大步数/轮数硬上限；② 工具白名单 + 权限；③ 每步可观测、可回放；④ 高风险动作加人工门；⑤ 评审器把关输出。

**Q：LangGraph / LangChain 这类框架解决什么？**
A：把「节点 + 边 + 状态 + 断点续跑 + 人审」这些 workflow 基础设施标准化。
本课用裸 TS 把这些概念手写一遍，理解了再用框架就是换皮。
